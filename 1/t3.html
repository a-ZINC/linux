Users & Groups

3–4 hours study
Beginner → Senior
Ubuntu 22.04 LTS
Prereq: T1 + T2

Linux is a multi-user operating system — every process, every file, and every action belongs to a user. The user and group model is the core security boundary of the entire OS. Get this wrong in production and you either lock out your own applications or open holes for attackers.

Understand UIDs, GIDs, and how the kernel uses them
Create, modify, and delete users and groups confidently
Explain how sudo works and configure /etc/sudoers safely
Read and write /etc/passwd, /etc/shadow, /etc/group
Switch users, run commands as another user
Create service accounts for applications (principle of least privilege)
Understand password policies, aging, and account locking
Debug "permission denied" and "sudo: user is not in the sudoers file"

Beginner Intuition
Section 01 — Simple Mental Model
🧠 Mental Model

Think of Linux as a secure office building. Each user has a badge with a unique ID number (UID). Each group is a department with its own ID (GID). Resources (files, devices, services) have locks that say: "badge 1001 can read, department 33 can write." The kernel checks your badge against the lock on every single operation. It never trusts names — it only ever reads the ID numbers.

There are three types of users you'll encounter:

UID 0 · /root · /bin/bash

UID 0superuserno restrictions
S

www-data, nobody, postgres...

UID 33–999 · no login shell · service accounts

system userno loginruns services
U

ubuntu, alice, bob...

UID 1000+ · /home/username · /bin/bash

regular userUID 1000+has home dir
✅ Try It Now

Run id right now. You'll see your UID, GID, and all groups you belong to. Run id root to see root's IDs. Run cat /etc/passwd | head -5 to see the first five user entries on your system.

Core Fundamentals
Section 02 — The user model, UIDs, GIDs, and sudo

UIDs and GIDs

Every user has a UID (User ID) — a non-negative integer. Every group has a GID (Group ID). The kernel identifies users exclusively by UID. Usernames are just human-readable labels stored in /etc/passwd and resolved by the C library.

UID RangeTypeExamples
0Root / superuserroot only. UID 0 = god mode — kernel bypasses all permission checks.
1–99Static system usersdaemon, bin, sys — defined by distro maintainers
100–999Dynamic system userswww-data (33), postgres (111), nobody (65534)
1000+Regular usersubuntu (1000), alice (1001), bob (1002)
65534nobodyUsed for unmapped UIDs (NFS, containers)

Primary group vs supplementary groups

Every user has exactly one primary group (stored in /etc/passwd, used as default GID for new files). They can also belong to many supplementary groups (stored in /etc/group). When you run id, you see all of them.

id ubuntu
# uid=1000(ubuntu) gid=1000(ubuntu) groups=1000(ubuntu),4(adm),24(cdrom),27(sudo),1001(docker)
#                  ↑ primary group   ↑ supplementary groups
# This user can: sudo (27), use docker (1001), read system logs (4:adm)

How sudo works

sudo (superuser do) allows a permitted user to run a command as root (or another user) after verifying their identity. It does NOT give you root access permanently — it elevates just for that one command, then drops back.

# Basic sudo usage
sudo apt update              # run command as root
sudo -u www-data whoami    # run as a specific user
sudo -i                     # open interactive root shell (login shell)
sudo -s                     # open root shell (non-login, keeps env)
sudo -l                     # list what THIS user is allowed to sudo
sudo -l -U alice            # list what alice is allowed to sudo (as root)

# sudo caches credentials for 15 minutes by default
sudo -k                     # invalidate cached credentials immediately

su — Switch User

su alice                   # switch to alice (needs alice's password)
su - alice                  # switch to alice with LOGIN shell (loads her env)
su -                        # switch to root (needs root password)
sudo su -                   # become root via sudo (uses YOUR password)
exit                        # return to previous user

# Difference: su vs sudo su -
# su - needs the TARGET user's password
# sudo su - needs YOUR password + sudo permission
# On cloud servers, root password is often not set — use sudo su -

Commands & Syntax
Section 03 — Every user and group management command

User management

# useradd — low-level user creation (manual setup needed)
useradd alice                          # create user (no home dir by default on some distros)
useradd -m alice                       # -m: create home directory
useradd -m -s /bin/bash alice         # set login shell explicitly
useradd -m -u 1500 -g developers alice # set UID and primary group
useradd -m -G sudo,docker alice       # add to supplementary groups
useradd -r -s /sbin/nologin myapp     # -r: system account, no login shell
useradd -M -s /sbin/nologin myapp     # -M: NO home directory (for services)

# adduser — high-level, interactive, Debian/Ubuntu preferred
adduser alice                          # interactive: asks password, full name, etc.
adduser alice sudo                     # add alice to sudo group
adduser --system --no-create-home myapp # create system account

# usermod — modify existing user
usermod -aG docker alice              # APPEND to group docker (-a is critical!)
usermod -aG sudo,adm alice           # append to multiple groups
usermod -s /bin/zsh alice            # change login shell
usermod -d /new/home alice           # change home directory
usermod -l newname alice             # rename username
usermod -L alice                      # LOCK account (prefix ! in shadow)
usermod -U alice                      # UNLOCK account
usermod -e 2025-12-31 alice          # set account expiry date

# userdel — delete user
userdel alice                          # delete user, KEEP home directory
userdel -r alice                       # delete user AND home directory AND mail spool
⚠️ Critical: usermod -aG vs usermod -G

usermod -G docker alice (without -a) replaces ALL supplementary groups with only docker. Alice loses sudo access, loses all other groups. Always use -aG to append. This mistake has locked engineers out of production servers.

Password management

passwd                               # change YOUR password
passwd alice                         # change alice's password (as root)
passwd -l alice                      # lock alice's password
passwd -u alice                      # unlock alice's password
passwd -d alice                      # delete password (allows passwordless login — dangerous)
passwd -e alice                      # expire password (force change on next login)
passwd -S alice                      # show password status

# chage — password aging policy
chage -l alice                       # list password aging info
chage -M 90 alice                    # max 90 days before forced change
chage -m 7 alice                     # min 7 days between changes
chage -W 14 alice                    # warn 14 days before expiry
chage -E 2025-12-31 alice           # account expires on date
chage -d 0 alice                     # force password change on next login

Group management

groupadd developers                  # create a new group
groupadd -g 2000 developers          # create with specific GID
groupmod -n newname developers       # rename group
groupdel developers                  # delete group (fails if it's anyone's primary group)
gpasswd -a alice developers          # add alice to developers
gpasswd -d alice developers          # remove alice from developers
gpasswd -M alice,bob,carol developers # set ALL members of group at once

# View group memberships
groups alice                         # list groups alice belongs to
id alice                             # detailed: uid, gid, all groups
getent group developers             # show group entry from /etc/group
getent passwd alice                  # show user entry from /etc/passwd

Querying users and sessions

whoami                               # print current effective username
id                                   # print uid, gid, all groups
who                                  # who is logged in right now
w                                    # who + what they're doing + idle time
last                                 # login history (reads /var/log/wtmp)
last alice                           # login history for alice only
lastb                                # failed login attempts (reads /var/log/btmp)
lastlog                              # last login for every account
finger alice                         # user info (if installed)

# newgrp — temporarily change active primary group
newgrp docker                        # new shell with docker as primary group
# New files created in this shell will have group "docker"
# NOTE: group changes via usermod -aG require logout/login to take effect
# OR: newgrp groupname to activate without logout

Configuring sudoers

visudo                               # ALWAYS use visudo — validates syntax before saving
# Never edit /etc/sudoers directly with vim/nano
# A syntax error in sudoers = NO ONE can use sudo = locked out

# /etc/sudoers syntax:
# user  host=(run_as)  command

root    ALL=(ALL:ALL) ALL        # root can do everything
alice   ALL=(ALL:ALL) ALL        # alice can do everything (admin)
%sudo   ALL=(ALL:ALL) ALL        # anyone in 'sudo' group
%wheel  ALL=(ALL:ALL) ALL        # anyone in 'wheel' group (RHEL/CentOS)

# Restrict to specific commands:
bob     ALL=(ALL) /usr/bin/systemctl restart nginx
deploy  ALL=(ALL) NOPASSWD: /usr/bin/rsync, /usr/bin/systemctl

# NOPASSWD — skip password prompt (common for automation)
ubuntu  ALL=(ALL) NOPASSWD: ALL   # cloud default — convenient, less secure

# Drop-in files (preferred over editing sudoers directly)
# Create /etc/sudoers.d/myapp  (mode 440, owned by root)

Important Files & Configs
Section 04 — The four key files that define identity on Linux
📁 The Four Identity Files

These four files together define every user and group on the system. Understanding their format lets you read, debug, and modify the user database manually when GUI tools aren't available.

/etc/passwd — User database (readable by all)

# Field: username pw UID GID GECOS home shell ubuntu:x:1000:1000:Ubuntu User:/home/ubuntu:/bin/bash www-data:x:33:33:www-data:/var/www:/usr/sbin/nologin nobody:x:65534:65534:nobody:/nonexistent:/usr/sbin/nologin
  • username — login name (1–32 chars, alphanumeric + underscore)
  • x — password placeholder (actual hash is in /etc/shadow)
  • UID — numeric user ID
  • GID — primary group ID
  • GECOS — full name / comment field
  • home — home directory path
  • shell — login shell (/usr/sbin/nologin = cannot log in)

/etc/shadow — Password hashes (root only: 640)

# user :hash :lastchg:min:max:warn:inactive:expire ubuntu:$6$rounds=5000$salt$longhash...:19738:0:99999:7::: alice:!:19738::::::: # ! = locked account myapp:*:19738::::::: # * = no password (system account)

The hash format: $algorithm$salt$hash — e.g., $6$ = SHA-512, $y$ = yescrypt (modern Ubuntu default).

/etc/group — Group database

# groupname : pw : GID : members (comma-separated) sudo:x:27:ubuntu,alice docker:x:998:ubuntu developers:x:2000:alice,bob,carol

/etc/sudoers and /etc/sudoers.d/

/etc/sudoers is the main sudo policy file. Never edit it directly — use visudo. The /etc/sudoers.d/ directory holds drop-in policy files, one per application or team. Files there must have mode 440 and be owned by root.

Other key files

FilePurposeAccess
/etc/login.defsDefault UID/GID ranges, password aging defaults, umaskroot
/etc/adduser.confadduser defaults (home dir skeleton, shell)root
/etc/skel/Skeleton files copied to new user's home on creationroot
/var/log/auth.logAll authentication events: logins, sudo, sshadm group
/var/log/secureSame but on RHEL/CentOS systemsroot
/etc/nsswitch.confName service switch — where to look up users (files, ldap, sssd)all

Under the Hood
Section 05 — How the kernel uses UIDs internally
🔬 Under the Hood

The kernel never uses usernames. It stores UIDs and GIDs everywhere: in inodes, in process tables, in socket connections. The username-to-UID mapping is purely a userspace convention resolved by the C library via NSS (Name Service Switch).

Real UID vs Effective UID

Every process has three UIDs that you must understand for security work:

UID TypeNameMeaning
RUIDReal UIDWho actually launched this process (your login UID)
EUIDEffective UIDWhat the kernel checks for permission decisions
SUIDSaved UIDAllows switching back to RUID after privilege drop
# Normal process (you running 'ls'):
# RUID=1000 (ubuntu)  EUID=1000 (ubuntu)  → no privilege

# sudo process:
# RUID=1000 (ubuntu)  EUID=0 (root)  → kernel sees root for permissions

# SUID binary like /usr/bin/passwd:
# RUID=1000 (you)  EUID=0 (root, because file is owned by root + SUID set)
# passwd can write to /etc/shadow (which only root can write)
# It drops back to RUID after finishing

PAM — Pluggable Authentication Modules

When you log in or use sudo, authentication goes through PAM — a stack of pluggable modules. PAM decides: is this password correct? Is the account locked? Is it within login hours? Has the password expired? This is configured in /etc/pam.d/.

# PAM stack for sudo (/etc/pam.d/sudo):
# auth   required  pam_env.so          → load environment variables
# auth   required  pam_unix.so         → check unix password
# auth   optional  pam_faillock.so     → count failed attempts

# View failed login lockout status:
faillock --user alice
# Reset lockout:
faillock --user alice --reset

nsswitch.conf and getent

# /etc/nsswitch.conf controls where user lookups go:
# passwd:   files systemd                 ← look in /etc/passwd first, then systemd
# group:    files systemd
# In enterprise: files ldap   or   files sss (SSSD for Active Directory)

# getent — query NSS databases (works for local AND LDAP/AD users)
getent passwd alice            # resolve user from any NSS source
getent group developers        # resolve group
getent passwd                  # list ALL users (including LDAP if configured)
# Use getent instead of cat /etc/passwd in enterprise environments

Real-World Usage
Section 06 — How teams and applications are set up in production
🏭 Real-World Patterns

These are the exact patterns used when provisioning production servers and deploying applications.

Creating a service account for an application

# Principle: every application runs as its own dedicated low-privilege user
# Never run apps as root. Never share user accounts between apps.

# Create a service account for an app called 'myapp'
useradd -r -M -s /sbin/nologin -d /opt/myapp -c "My App Service" myapp
# -r: system account (UID < 1000)
# -M: no home directory
# -s /sbin/nologin: cannot log in interactively
# -d /opt/myapp: home set to app dir for reference

# Set ownership of app files:
chown -R myapp:myapp /opt/myapp
chmod -R 750 /opt/myapp

# Verify: can this account log in? (should fail)
su -s /bin/bash myapp -c "whoami"

Setting up a dev team with shared access

# Create a team group
groupadd -g 2000 devteam

# Add developers to the group
usermod -aG devteam alice
usermod -aG devteam bob
usermod -aG devteam carol

# Create shared project directory
mkdir /opt/project
chown root:devteam /opt/project
chmod 2775 /opt/project          # setgid: new files inherit devteam group

# Give devteam specific sudo privileges (deploy only):
# /etc/sudoers.d/devteam
%devteam ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart myapp, /usr/bin/journalctl

Onboarding a new engineer (full checklist)

adduser newengineer               # create with home dir, interactive
usermod -aG sudo newengineer      # grant sudo
usermod -aG docker,devteam newengineer
chage -d 0 newengineer            # force password change on first login
chage -M 90 -W 14 newengineer   # 90-day password policy with 14-day warning
# Add SSH public key:
mkdir -p /home/newengineer/.ssh
chmod 700 /home/newengineer/.ssh
echo "ssh-ed25519 AAAA..." >> /home/newengineer/.ssh/authorized_keys
chmod 600 /home/newengineer/.ssh/authorized_keys
chown -R newengineer:newengineer /home/newengineer/.ssh

Common Beginner Mistakes
Section 08
  • usermod -G without -a: Replaces all supplementary groups. Use usermod -aG always. This is the most common user-management mistake and it causes lockouts.
  • Group membership changes need re-login: Adding a user to a group takes effect in new sessions only. Existing terminal sessions still use the old groups. Use newgrp groupname or log out and back in.
  • Confusing su - and su: su alice switches user but keeps your current environment. su - alice gives a full login shell with alice's environment. Almost always use su -.
  • Editing /etc/sudoers directly: Always use visudo. It validates the file before saving. A syntax error in sudoers means nobody can sudo — emergency recovery from console needed.
  • Creating users with the same UID: If two entries in /etc/passwd share a UID, the kernel treats them as the same user. Files owned by one are accessible to the other. UIDs must be unique.
  • Not setting a shell for service accounts: Forgetting -s /sbin/nologin when creating a service account means it can potentially log in if someone sets a password.
🏗️
Common Production Mistakes
Section 09
  • Running applications as root: If the app is compromised, the attacker has root. Every application should run as a dedicated user with only the access it needs — nothing more.
  • NOPASSWD: ALL in sudoers for all users: Common on cloud images for convenience (e.g., default Ubuntu AWS AMI). Acceptable if SSH key auth is enforced; dangerous if password auth is also allowed.
  • Not auditing who's in the sudo group: getent group sudo. If a contractor account or old employee account is still in sudo, that's an open door.
  • Shared service accounts between applications: If www-data runs both nginx and your PHP app, a PHP exploit can read nginx configs. Separate users per app.
  • Forgetting to delete user home directory on offboarding: userdel alice removes the account but leaves /home/alice with all its SSH keys and scripts. Use userdel -r or manually clean up.
  • Password expiry breaking service accounts: Applying a password policy (chage -M 90) to service accounts causes them to "expire" — their processes then fail to authenticate. Service accounts should have chage -M -1 (no expiry).

Debugging Workflow
Section 10 — Diagnosing user and permission problems
🛠 Debugging Checklist

When you see "permission denied" or "not in sudoers", work through this order.

# "sudo: user is not in the sudoers file"
getent group sudo              # is the user in the sudo group?
sudo -l -U alice               # what can alice sudo? (run as root)
cat /etc/sudoers.d/*          # check drop-in rules
# Fix: usermod -aG sudo alice  (then they must log out/in)

# "Permission denied" on a file owned by root
id                             # what user/groups am I?
ls -la thefile                 # who owns it, what permissions?
groups                         # am I in the right group?
# If I should have access via group, check if I logged in after being added

# "This account is currently not available" (nologin shell)
getent passwd username         # check the shell field (last field)
# If /usr/sbin/nologin — this is by design for service accounts
# To run a command AS this user: sudo -u myapp /opt/myapp/test.sh
# To debug: sudo -u myapp -s /bin/bash  (override shell just for this session)

# Account locked
passwd -S alice                # LK = locked, PS = has password, NP = no password
chage -l alice                 # check if password/account expired
grep alice /etc/shadow        # ! prefix on hash = locked
# Fix: passwd -u alice  or  usermod -U alice

# Check recent auth events
tail -50 /var/log/auth.log    # Ubuntu: all authentication events
grep "alice" /var/log/auth.log | tail -20
journalctl _COMM=sudo          # all sudo invocations (systemd systems)

Security Considerations
Section 11 — The most security-critical Linux topic
🔐 This is the most security-critical topic in Phase 1

More real-world breaches involve misconfigured user accounts, excessive sudo privileges, or service accounts running as root than almost any other Linux misconfiguration. Get this right.

  • Principle of Least Privilege: Every user and service account should have only the minimum permissions required to do its job. Nothing more.
  • Audit sudoers regularly: grep -r NOPASSWD /etc/sudoers /etc/sudoers.d/ — every NOPASSWD entry is a risk. Justify each one.
  • Audit sudo group membership: getent group sudo wheel. Remove accounts that no longer need elevated access.
  • Disable root SSH login: In /etc/ssh/sshd_config: PermitRootLogin no. Users log in as themselves, then sudo.
  • Lock unused accounts: Former employees, testing accounts, default vendor accounts. usermod -L username or chage -E 1 username.
  • Watch for UID 0 accounts: awk -F: '($3 == 0) {print}' /etc/passwd — there should only ever be one: root. Any other UID 0 account is a backdoor.
  • Use SSH keys, not passwords: Disable password authentication in sshd_config for all interactive logins. SSH keys are far harder to brute-force than passwords.
# Security audit commands — run these on any new server
awk -F: '($3 == 0) {print}' /etc/passwd     # UID 0 accounts (should be root only)
awk -F: '($2 == "" ) {print}' /etc/shadow    # accounts with no password (NP)
grep -r "NOPASSWD" /etc/sudoers /etc/sudoers.d/
getent group sudo wheel                         # who has sudo?
lastb | head -20                                # recent failed login attempts
last | grep "still logged in"                  # who's currently in?

Performance Considerations
Section 12
  • NSS lookup latency: On systems using LDAP or Active Directory via SSSD, user lookups (id, file ownership resolution) make network calls. A misconfigured or unreachable LDAP server causes hangs on every ls -la. Local caching via SSSD mitigates this.
  • Large groups slow group membership checks: A group with thousands of members (/etc/group line very long) can slow down permission checks. Use LDAP nested groups for large organisations instead of flat local groups.
  • PAM modules add login latency: Complex PAM stacks (LDAP + OTP + audit logging) can add 1–3 seconds to every SSH login. Profile with time ssh user@host exit.
  • sudo credential caching: The default 15-minute sudo timestamp cache means users don't re-authenticate frequently. Reduce with Defaults timestamp_timeout=5 for higher security environments.

Best Practices — Senior Level
Section 13

How a senior engineer thinks about users and groups

  • They create a service account for every application. The account has no login shell, no home directory, and owns only what the app needs. This is non-negotiable.
  • They use visudo exclusively and always test sudoers changes by opening a second terminal before closing the first.
  • They never grant NOPASSWD: ALL unless there is a documented, justified reason. Automation gets NOPASSWD for specific commands only.
  • They run sudo -l as a habit when landing on an unfamiliar server — know your privileges immediately.
  • They script user provisioning (Ansible, Terraform) rather than doing it manually — consistent, auditable, repeatable.
  • They know that group membership changes require re-login, and they communicate this clearly to engineers who report "it still doesn't work" after being added to docker.
  • They check /var/log/auth.log as part of any incident investigation — auth logs tell you exactly who did what and when.
# Sudoers drop-in best practice
# File: /etc/sudoers.d/deploy
# Mode: 440  Owner: root:root

# Deployment user: restart specific services only
deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart myapp, \
                            /usr/bin/systemctl status myapp, \
                            /usr/bin/journalctl -u myapp

# Create it safely:
echo 'deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart myapp' > /tmp/deploy_sudo
visudo -c -f /tmp/deploy_sudo    # validate syntax before installing
cp /tmp/deploy_sudo /etc/sudoers.d/deploy
chmod 440 /etc/sudoers.d/deploy

Real Incident Scenarios
Section 14
🚨 Production Incident — Simulation

The sudo Lockout

Scenario: An engineer edits /etc/sudoers directly with vim, saves a file with a typo. All subsequent sudo commands fail with "syntax error". Nobody can sudo.

Symptoms: sudo: /etc/sudoers is owned by uid 0, not uid 0 or sudo: parse error in /etc/sudoers

Recovery (requires root console access):

# Option 1: If you have another terminal still open with sudo cached:
sudo visudo                    # fix the error while cache is valid

# Option 2: Boot to recovery mode (single user / emergency.target)
# At the root prompt:
visudo                         # fix the file

# Option 3: Cloud provider console/serial access as root
pkexec visudo                  # pkexec uses PolicyKit, not sudoers

Prevention: Always use visudo. Always keep a second terminal open when modifying sudoers. Never edit sudoers on a server you can only reach via SSH.

🚨 Production Incident — Simulation

The Group Change That Broke Docker

Scenario: New developer added to docker group. They run docker ps and get "permission denied on /var/run/docker.sock". They report it's broken. You re-check and confirm they are in the group.

Root cause: usermod -aG docker alice takes effect only in new login sessions. Alice's existing terminal session still has the old group list.

Fix:

# Alice runs in her existing terminal:
newgrp docker       # activates group for current shell
# Or: log out and log back in
# Verify:
id                  # docker should now appear in groups list

1Find all UID 0 accounts: awk -F: '($3==0){print $1}' /etc/passwd. There should be only root.

Break It & Fix It
Section 16
🧪 Break It & Fix It

All exercises are safe and recoverable. Do them in order — each teaches a specific debugging skill.

  • Break 1 — Remove all groups (without -a): Create testuser2. Add them to a test group. Then run sudo usermod -G anothergroup testuser2 (no -a). Check id testuser2. Observe groups lost. Fix: usermod -aG original_group testuser2.
  • Break 2 — Lock yourself out of sudo temporarily (SAFE): Create a test sudoers file with a deliberate syntax error in /tmp/test_sudoers. Run visudo -c -f /tmp/test_sudoers to see the error. This shows you why visudo validation matters without actually breaking anything.
  • Break 3 — Expired password: Create testuser3. Force expire their password: sudo chage -d 0 testuser3. Try to log in — they should be forced to change password. Fix: set a new password as that user.
  • Break 4 — nologin shell: Create testuser4 with nologin shell: sudo useradd -s /sbin/nologin testuser4. Attempt sudo su - testuser4. Observe the "account not available" message. Fix: sudo usermod -s /bin/bash testuser4.
  • Break 5 — Group membership not active: Add yourself to a new group (sudo groupadd testgrp && sudo usermod -aG testgrp $USER). Try to access a resource that requires testgrp. Fail. Fix using newgrp testgrp. Verify with id.

How a senior engineer thinks about users and groups

Every user account is a security boundary. Senior engineers treat user and group design like network firewall rules — default deny, explicit allow. A web application does not need to read /etc/shadow. A deployment script does not need to restart every service on the system. Each account gets exactly the minimal set of permissions for its job, applied through the combination of file ownership, group membership, and scoped sudoers rules. They automate provisioning so it is consistent and auditable. And they always check the auth log during incidents — if something unexpected happened on a Linux system, /var/log/auth.log was watching.
